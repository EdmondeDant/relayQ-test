[OPEN] image-edit-failure

## 症状
- 在线体验中的“图片编辑”上传了一张本地图片后执行失败。
- 页面只显示“图片处理失败，本次不应扣费。”，没有展示真实错误原因。
- 用户怀疑是“桌面上的原图”本身或本地文件路径导致。

## 期望
- 能明确区分是本地文件读取问题、内容审核/模型拒绝，还是异步任务链路本身出错。

## 当前可证伪假设
1. 本地文件路径不是问题，浏览器已经把桌面图片转成了 `blob:`/`data:`，失败发生在上传之后。
2. 这张图片或提示词命中了内容审核/上游策略，后端返回了真实拒绝原因，但前端只显示了通用失败文案。
3. 新的 Playground 异步任务提交成功了，但前端轮询/错误透传没有把 `task.error_message` 正确显示出来。
4. 图片编辑请求体构造有误，导致上游返回参数错误而不是图片内容问题。
5. 异步任务 worker 执行正常，但资产落库或结果恢复失败，被前端误判成“图片处理失败”。

## 调试计划
1. 查前端图片文件读取和异步任务提交逻辑。
2. 查后端任务执行/状态更新是否记录了真实错误。
3. 必要时补最小埋点，记录提交、轮询和失败原因。

## 已完成
- 已启动调试会话 `image-edit-failure`，日志文件为 `.dbg/trae-debug-log-image-edit-failure.ndjson`。
- 已补前端埋点：记录图片编辑任务提交、任务创建、轮询状态与失败信息。
- 已补后端埋点：记录异步图片编辑请求开始、上游报错、返回空图片等关键节点。
- 已拿到第一轮运行时证据：日志里出现了“前端开始提交编辑任务”，但没有“任务创建成功”，说明失败点发生在 `POST /playground/jobs` 之前或当场返回错误。
- 已修第一层错误透传：前端不再把普通对象错误一律降级成“图片处理失败，本次不应扣费。”。
- 用户复测后已看到 `Request failed with status code 500`，说明第一层透传生效，但还需继续深入展示后端返回体中的真实 message。
- 已补第二层错误透传：优先展示 `response.data.message / error.message / details / 原始 body`。
