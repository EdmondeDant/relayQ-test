## 2. 官方行为澄清

经过对 Xiaomi MiMo 官方文档和官方 Token Plan 上游的直接核对，可以确认：

- `https://token-plan-cn.xiaomimimo.com/v1` 是官方 Token Plan OpenAI 兼容入口
- `tp-*` 是官方 Token Plan API Key 形态
- `mimo-v2.5` / `mimo-v2-omni` 的音频理解能力，不等于 `mimo-v2.5-asr` 的入参约束

本次最终确认的关键点是：

- `mimo-v2.5-asr` 在实测中要求 `input_audio.data` 为
  - 有效 base64 数据，或
  - 带 MIME 前缀的 data URL
- 当请求直接传普通音频 URL 时，上游会返回参数错误
- 当请求传 `data:audio/wav;base64,...` 这类 data URL 时，上游可正常识别

结论：

- 不是“小米官方上游异常”
- 真正需要修的是网关对 ASR 音频入参的兼容处理

## 3. 本次代码变更

### 3.1 MiMo 音频模型接入

- `mimo-v2-pro`
- `mimo-v2-omni`
- `mimo-v2-tts`

### 3.2 MiMo 音频模型 raw chat 分流

由于部分 MiMo 音频模型不适合走当前的 Chat Completions -> Responses 转换路径，本次已对以下模型强制走 raw Chat Completions 转发：

- `mimo-v2.5-asr`
- `mimo-v2.5-tts`
- `mimo-v2.5-tts-voiceclone`
- `mimo-v2.5-tts-voicedesign`
- `mimo-v2-tts`

这样可以保持请求体尽量贴近 MiMo 官方 OpenAI 兼容接口的原始要求，减少中间协议转换对音频字段的破坏。

### 3.3 ASR 自动规范化兼容层

本次新增了一个非常窄范围的兼容层，仅作用于：

- 模型：`mimo-v2.5-asr`

兼容规则如下：

- 如果 `messages[].content[].type = input_audio`
- 且 `input_audio.data` 是裸 base64
- 且 `input_audio.format` 可识别为 `wav/mp3/pcm`

则网关会自动把：

```text
UklGRg==
```

改写为：

```text
data:audio/wav;base64,UklGRg==
```

以下情况保持原样，不做改写：

- 已经是 `data:` URL
- 已经是 `http://` 或 `https://` URL
- `format` 无法识别

这一步的目的，是让调用方不需要自己记住 `data:audio/...;base64,...` 这个细节。
