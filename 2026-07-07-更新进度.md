# 2026-07-07 更新记录：零售 Grok Key、接口文档、客户端安装与生产隔离

## 1. 更新背景

本次更新围绕两个目标展开：

1. 为零散客户、小额试用、闲鱼/二手小店售卖 token 场景增加独立的 Grok 零售 Key 能力。
2. 在不影响现有生产 API Key、现有 `/v1` 主链路、现有每日统计和用量体系的前提下，提供独立管理、独立调用、独立用量查询和独立接口说明。

本次改动坚持“零影响生产优先”：零售功能使用独立路由、独立表、独立鉴权、独立前端页面，不接入现有生产 API Key 表和主用量统计链路。

## 3. 零售调用接口

### 3.1 OpenAI 兼容 Base URL

零售客户应配置：

`base_url = https://www.relayq.top/retail/v1`

鉴权：

`Authorization: Bearer YOUR_RETAIL_GROK_KEY`

零售 Key 前缀：

`rgk-`

### 3.2 模型列表接口

新增 OpenAI 兼容模型列表：

`GET /retail/v1/models`

返回格式：

```json
{
  "object": "list",
  "data": [
    { "id": "grok-4.3", "object": "model", "owned_by": "xai", "created": 0 },
    { "id": "grok-imagine-image", "object": "model", "owned_by": "xai", "created": 0 },
    { "id": "grok-imagine-image-quality", "object": "model", "owned_by": "xai", "created": 0 },
    { "id": "grok-imagine-video", "object": "model", "owned_by": "xai", "created": 0 }
  ]
}
```

该接口加入零售非消耗类鉴权放行，不扣额度。有效零售 Key 可用它刷新模型列表。

相关文件：

- `backend/internal/server/routes/retail_grok.go`
- `backend/internal/handler/retail_grok_gateway_handler.go`
- `backend/internal/server/middleware/retail_grok_key_auth.go`

### 3.3 文本 / 多模态识图

接口：

`POST /retail/v1/chat/completions`

支持模型：

- `grok-4.3`

用途：

- 文本推理。
- 图片理解。
- 多模态识图。

计费：

- 记录 input tokens。
- 记录 output tokens。
- 累加 `token_used_total`。

### 3.4 文生图

接口：

`POST /retail/v1/images/generations`

支持模型：

- `grok-imagine-image`
- `grok-imagine-image-quality`

主要参数：

- `prompt`
- `n`
- `aspect_ratio`
- `resolution`
- `response_format`

计费：

- 按 `ImageCount` 累加 `image_used_total`。

### 3.5 图生图 / 图片编辑

接口：

`POST /retail/v1/images/edits`

支持模型：

- `grok-imagine-image`
- `grok-imagine-image-quality`

支持：

- 单图 `image`。
- 多图 `images`。
- 图片 URL。
- base64 data URL。

计费：

- 按 `ImageCount` 累加 `image_used_total`。

### 3.6 文生视频 / 图生视频

接口：

`POST /retail/v1/videos/generations`

其他视频接口：

- `POST /retail/v1/videos/edits`
- `POST /retail/v1/videos/extensions`
- `GET /retail/v1/videos/:request_id`

支持模型：

- `grok-imagine-video`

主要参数：

- `prompt`
- `duration`
- `aspect_ratio`
- `resolution`
- `image`
- `reference_images`

计费：

- 成功提交视频任务后累加 `video_used_total`。
- 视频轮询接口不扣额度。

## 5. 今日修复的问题

### 5.1 管理端生成接口 404

问题：

- 管理端点击生成零售 Key 时返回 404。

原因：

- 后端旧进程未重启，新路由未加载。

处理：

- 重启后端后确认 `/api/v1/admin/retail-grok/batch-generate` 生效。

### 5.2 客户查询页无显示

问题：

- 客户查询页输入 Key 后没有显示。

原因：

- 前端 dev server 请求 `/retail/v1/usage` 时被 Vite fallback 返回 `index.html`，HTTP 200 但不是 JSON。

处理：

- 新增 `/api/v1/retail-grok/usage` 兼容查询路由。
- 前端查询改走 `/api/v1/retail-grok/usage`。
- 前端增加响应校验，避免 HTML 假成功。

### 5.3 客户接口说明不完整

问题：

- 文档缺少多模态识图、官方图片参数、图生图格式、视频分辨率和时长参数。

处理：

- 补齐 Grok 4.3 多模态识图示例。
- 补齐文生图参数。
- 补齐图生图/多图编辑示例。
- 补齐文生视频和图生视频示例。
- 明确 OpenAI 兼容 `base_url`。

### 5.4 CSV 导出不是全部 Key

问题：

- 原 CSV 只导出当前生成批次。

处理：

- 点击导出时重新请求后端最多 10000 条零售 Key。
- 导出按钮改为“导出全部 CSV”。

### 5.5 删除按钮报错

问题：

- 管理页点击删除可能报错。

原因：

- 原实现物理删除 `retail_grok_keys`，对有用量日志或需要审计的 Key 不合适。

处理：

- 改成软删除：`status = 'deleted'`。
- `GetByID`、`GetByKey`、`List` 均排除 `deleted`。
- 被删除 Key 不再可用于零售接口。
- 历史用量日志保留。

### 5.6 视频调用不扣次数

问题：

- 视频接口成功调用后 `video_used_total` 不增加。

原因：

- 视频成功后传入 `nil` result，原计费构造器在 `result == nil` 时提前返回，导致 `video_count = 0`。

处理：

- 对视频端点先设置 `video_count = 1`，再处理 result。

相关文件：

- `backend/internal/service/retail_grok_gateway_service.go`

### 5.7 文本 token / 图片次数可能不落库

问题：

- 文本和图片调用成功，但用量页看不到增加。

原因：

- 零售计费写库使用 `c.Request.Context()`，而写库发生在响应转发完成之后。如果客户端连接关闭或流式响应结束，context 可能被取消，导致写库失败。
- 原先 `RecordUsage` 错误被静默吞掉。

处理：

- 零售计费写库改为独立 `context.Background()` + 10 秒超时。
- 保留 request id / client request id 作为日志上下文。
- `RecordUsage` 失败时写日志 `retail_grok.record_usage_failed`。

相关文件：

- `backend/internal/handler/retail_grok_gateway_handler.go`

### 5.8 `rgk-` 误调普通 `/v1/*` 必须失败

问题：

- 零售 Key 不能误进入普通生产接口。

处理：

- 普通 API Key 鉴权遇到 `rgk-` 前缀直接返回 401。
- 错误信息提示必须使用 `/retail/v1`。

相关文件：

- `backend/internal/server/middleware/api_key_auth.go`

### 5.9 零售模型列表刷不出来

问题：

- OpenAI SDK / 客户端配置 `base_url = https://www.relayq.top/retail/v1` 后通常会请求 `/models`。
- 之前没有 `GET /retail/v1/models`。

## 4. 已定位并处理过的问题

### 4.1 账号测试正常，模型测试/聊天失败

排查结论：

- 账号测试和正式聊天并不是同一条链路
- 账号测试走 `AccountTestService`
- 模型测试/聊天走 API Key 网关与调度链路

已经确认并修过的几个问题：

- `xai` 文本请求最开始被错误路由进通用 `Gateway`
- `xai` 的 `403` 处理过于激进，第一次失败就会被打成 `error`
- `OpenAIAccountScheduler` 中残留 `IsOpenAI()` 判断，把 `xai` 账号过滤掉
- 调度快照在部分重建路径里漏掉 `xai`
- 快照查询时错误传入空平台，导致 `no available accounts`

这些问题解释了此前的：

- `All available accounts exhausted`
- `Service temporarily unavailable`

### 4.2 `xai OAuth` 被错误当成 OpenAI Codex OAuth

后续继续排查后，确认还有更深层问题：

- `xai OAuth` 在正式转发时被错误发往：

```text
https://chatgpt.com/backend-api/codex/responses
```

- 同时还复用了 ChatGPT/Codex 专属逻辑，例如：
  - `chatgpt-account-id`
  - Codex body transform
  - Codex Responses 桥接

这会导致：

- 账号测试成功
- 但聊天/模型测试在正式请求时仍报：
  - `502 Bad Gateway`
  - `Upstream request failed`

本轮已经做过若干修补：

- 将 `xai OAuth` 从部分 `chatgpt.com` 转发分支中拆出
- 尝试让 `xai OAuth` 走 `api.x.ai`
- 尝试把 `xai OAuth` 的聊天流量改为原生 `chat/completions` 直转

但截至本轮结束，还没有完成最终的结构化重构与最终验证闭环。

## 5. 外部资料调研结论

本轮后半段已停止盲猜，转而核对官方文档和 GitHub 成熟实现。核心结论如下。

### 5.1 官方文档结论

xAI 官方推理接口是：

- `https://api.x.ai/v1/chat/completions`
- `https://api.x.ai/v1/responses`

鉴权是普通：

- `Authorization: Bearer <token>`

不是 OpenAI 的：

- `https://chatgpt.com/backend-api/codex/responses`

参考：

- `https://docs.x.ai/developers/rest-api-reference/inference/chat`

### 5.2 成熟开源实现的共识

查到的成熟参考包括：

- Hermes Agent
- 9router
- AxonRouter
- opencode
- grok-oauth-proxy

这些实现的共同点是：

- `xai OAuth` 被当成独立 provider
- OAuth 与请求转发都不复用 OpenAI Codex 的特殊链路
- 请求直接打 `api.x.ai/v1`
- 不注入 `chatgpt-account-id`
- 不走 ChatGPT internal `codex/responses`
- refresh 策略通常为：
  - 过期前主动 refresh
  - 401 / bad credentials 时再 reactively refresh 一次

### 5.3 关于 `403` 的关键现实结论

外部资料还确认了一个很关键的现实：

- xAI OAuth 登录成功，不代表推理一定有权限
- 某些 SuperGrok / X Premium+ 账号即使登录成功，仍可能在 API 推理阶段返回 `403`
- 成熟实现通常会把这类错误提示为：
  - OAuth 已连接
  - 但 API access denied / tier / allowlist 限制

而不是直接粗暴打坏账号。

## 6. 当前最可信的架构结论

本项目目前最大的问题，不是单个 if 写错，而是：

- `xai OAuth` 被混入了 `openai oauth / codex oauth` 的实现体系

而根据官方文档和成熟实现，正确做法应是：

- `xai OAuth` 独立成一条 provider 链路

至少应独立以下几层：

- 授权端点与 token exchange
- refresh 逻辑
- upstream base URL
- 请求头注入
- body transform
- 401/403 的错误分类
- 调度时的临时不可调度/永久错误策略

换句话说：

- Grok 文本接口是 OpenAI-compatible
- 但 Grok OAuth 不是 OpenAI Codex OAuth

### 17.5 图片/视频参数对齐 xAI 官方接口（2026-07-05）

目标：按 xAI/Grok Imagine 官方接口尽量 1:1 对齐媒体参数，减少中转层误转换导致的 422/invalid_argument，并释放模型完整能力。

官方依据：

- 图片生成 `/v1/images/generations`：`prompt`、`model`、`n`、`aspect_ratio`、`resolution`（`1k`/`2k`）、`response_format`（`url`/`b64_json`）、`quality`。`size` 官方标注为不支持。
- 图片编辑 `/v1/images/edits`：JSON body，`image: {url, type}`；OpenAI SDK multipart `images.edit()` 并非 xAI 官方格式。
- 视频生成 `/v1/videos/generations`：`prompt`、`model`、`duration`、`aspect_ratio`、`resolution`（`480p`/`720p`，`1080p` 仅部分模型场景支持）、`image`、`reference_images`。
- 图片转视频：官方格式为 `image: {url: ...}`。
- 参考图视频：官方格式为 `reference_images: [{url: ...}]`，最多 7 张，且参考图模式最大 10 秒。

已完成：

- 图片 Chat Completions 生图桥接支持并透传：
  - `n`（限制到官方最大 10）
  - `aspect_ratio` / `aspectRatio`
  - `resolution`（规范化为 `1k`/`2k`）
  - `quality`
  - `user`
  - 支持从 `image_options`、`providerOptions.xai`、`provider_options.xai` 读取参数
  - 多图返回时包装为多行 Markdown 图片，仍保持标准 Chat Completions/stream SSE 输出
- `/v1/images/generations` 继续保留官方 JSON 参数透传：
  - `aspect_ratio`、`resolution`、`quality` 等会进入上游 body
- Responses/Image tool 桥接补充 `aspect_ratio`、`resolution` 透传
- 视频兼容层支持：
  - `aspect_ratio` / `aspectRatio`
  - `resolution`，并规范化 `HD -> 720p`、`SD -> 480p`、`1080 -> 1080p` 等
  - OpenAI/Sora `size` 自动转换为官方 `aspect_ratio` + `resolution`，随后移除 `size`，避免 xAI 官方接口不支持字段
  - `input_reference.image_url` 默认转换为官方 image-to-video：`image.url`
  - `providerOptions.xai.mode=reference-to-video` 时转换为官方 reference-to-video：`reference_images: [{url: ...}]`
  - `providerOptions.xai.referenceImageUrls` / `provider_options.xai.referenceImageUrls` 转为官方 `reference_images`
  - 规范已有 `reference_images` 中的字符串或 `{image_url}` 为官方 `{url}`
