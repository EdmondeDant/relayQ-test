## 十二、7 月 14 日补充更新（二）— 一键安装国内网络中转与自动回退

### 1. 背景问题

- 很多客户在使用项目首页的一键安装命令安装 `OpenClaw` / `Hermes` 时，会在下载 `Node.js`、`Git`，以及 Hermes 官方安装脚本和仓库快照过程中，因为国内网络访问上游站点不稳定而失败。
- 用户诉求是：
  - 不要求客户手动替换源
  - 不要求客户修改命令
  - 继续保持一条命令自动安装

### 2. 本次改造目标

- 将 Windows 一键安装链路改造成：
  - 优先走 RelayQ 自己站点的 `/downloads/...` 中转地址
  - 中转失败时自动回退到官方源
- 保证客户端用户仍然只执行原有一键安装命令，不需要任何手工干预。

### 3. OpenClaw Windows 安装链路增强

- 在 `OpenClaw` Windows 安装脚本里新增依赖预检查：
  - 自动检测 `Node.js`
  - 自动检测 `Git`
- 若缺失则自动下载安装：
  - 先尝试 RelayQ 自己站点中转地址
  - 失败后回退到官方地址
- 依赖就绪后，再继续执行 OpenClaw 官方安装脚本和原有 onboarding 流程。

### 4. Hermes Windows 安装链路增强

- 在 `Hermes` Windows 安装脚本里新增依赖预检查：
  - 自动检测 `Node.js`

## 十三、7 月 22 日补充更新（三）— Playground 异步图片链路修复

### 3. 运行时调试与证据采集

### 4. 根因一：异步 worker 内部回环地址错误

- 新任务最初会一直停在 `pending`，进一步抓日志后确认并不是上游图片接口本身挂掉，而是后端 worker 自己发内部请求时打错了地址。

  - 将资产相对路径解析也改为复用同一 base URL

  - `backend/internal/service/playground.go`

### 5. 根因二：任务状态回写 SQL 参数类型冲突

- 在修复回环地址后，浏览器现场验证已经能产出结果图，但任务与创作记录状态仍然停留在 `pending`。

  - 同一个占位符同时被用于 `status = $3`
  - 又被用于 `CASE WHEN $3 IN (...)`
- 该问题会导致：
  - `running` 状态更新失败
  - `succeeded` 状态更新失败
  - 数据库里有产物，但任务状态不推进
- 本轮修复方式：
  - 在仓储层显式拆分 `setStarted` 和 `setCompleted` 两个布尔参数
  - 避免在 SQL 中复用同一个文本参数做集合判断
- 涉及文件：

## 十四、7 月 22 日补充更新（四）— priority 模式下 cache write 计费对齐

### 1. 背景问题

- 在本地多次调整后的计费链路中，`priority` service tier 已经能对：
  - 输入 token
  - 输出 token
  - cache read
  使用高优先级单价。
- 但 `cache write` 仍沿用普通单价，没有切到 `priority` 对应价格。
- 这会导致启用 `priority` 的请求中，只要存在 `cache_creation_tokens`，客户实际扣费会低于原版 GitHub 逻辑。

### 2. 根因确认

- 本地 `ModelPricing` 结构缺少：
  - `CacheCreationPricePerTokenPriority`
- 本地动态价格解析 `PricingService` 也没有读取：
  - `cache_creation_input_token_cost_priority`
- 因此即使上游定价数据提供了 `priority cache write` 单价，计费核心也无法接收到这个字段。

### 3. 本轮修复内容

- 在 `billing_service.go` 中为 `ModelPricing` 增加：
  - `CacheCreationPricePerTokenPriority`
- 在 `usePriorityServiceTierPricing()` 中把 `cache write priority` 纳入显式优先级价格判断。
- 在 `computeTokenBreakdown()` 中新增 `cacheCreationPrice` 变量：
  - 普通模式下使用 `CacheCreationPricePerToken`
  - `priority` 模式下若存在显式优先级价格，则改用 `CacheCreationPricePerTokenPriority`
- 将 `computeCacheCreationCost()` 调整为显式接收当前应使用的 cache write 单价，避免仍从普通字段取值。

### 4. 动态价格与渠道覆写同步对齐

- 在 `pricing_service.go` 中补齐：
  - `CacheCreationInputTokenCostPriority`
  - `cache_creation_input_token_cost_priority` JSON 字段解析
- 在 `GetModelPricing()` 中将动态价格里的 `cache_creation_input_token_cost_priority` 映射到：
  - `CacheCreationPricePerTokenPriority`
- 在 `GetModelPricingWithChannel()` 中，让渠道配置里的 `CacheWritePrice` 同时覆盖：
  - `CacheCreationPricePerToken`
  - `CacheCreationPricePerTokenPriority`
  - `CacheCreation5mPrice`
  - `CacheCreation1hPrice`

### 5. 定向测试补充

- 新增并更新了以下测试覆盖：
  - `priority` 下显式 `cache write` 高优价格是否生效
  - 动态价格字段是否成功映射进计费模型
  - 渠道 `CacheWritePrice` 是否同时覆盖普通价与 `priority` 价

### 6. 本轮验证结果

- 已通过定向测试：
  - `go test ./internal/service -run "Priority|CacheWrite|GetModelPricing"`
- 本轮修改未引入新的本地诊断错误。

### 7. 涉及的关键文件补充

- `backend/internal/service/billing_service.go`
- `backend/internal/service/pricing_service.go`
- `backend/internal/service/billing_service_test.go`

### 8. 结论

- 本轮已将 `priority cache write` 计费逻辑对齐到 GitHub 原版思路。
- 修复后，`priority` 模式下的缓存写入费用将不再错误沿用普通单价。
- 本轮仅修复这一处计费偏差，没有顺带修改其他长上下文、fallback 或图片计费策略。
