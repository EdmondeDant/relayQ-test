# 2026-07-23 更新进度

## 本轮补充修复

### 一、修复 Playground 视频成功但未扣费

- 排查确认：Playground 的 `grok-imagine-video` / `video` 异步任务在成功后，原先只完成了任务状态推进与视频资产落库，没有进入正式 usage / billing 记账链路。
- 已补齐视频任务成功后的 usage 记录与统一 billing apply 逻辑。
- 现在视频任务完成后会：
  - 生成独立 usage log
  - 使用 `playground-video:<request_id>` 作为幂等 request_id
  - 调用统一扣费逻辑，避免只生成视频不扣费

### 二、优化创作记录加载速度

- 排查确认：创作记录接口本身耗时不高，主要瓶颈来自前端列表页加载后对每条记录执行媒体预取、鉴权下载与 blob 转换。
- 已调整 `loadCloudRecords()`：
  - 先只加载记录元数据
  - 不再在列表阶段对全部记录执行全量 hydrate / fetch
  - 媒体改为在真正恢复或查看时按需读取
- 预期效果：
  - 创作记录页面首屏明显更快
  - 额外媒体请求显著减少
  - 浏览器控制台中由预取链路带来的噪音报错同步下降

### 三、修复 Playground 对话助手的模型筛选误伤

- 之前为规避不可用模型，曾错误按 `platform === anthropic / antigravity` 将模型整体排除出对话助手，误伤 Claude 可用模型。
- 已改为仅排除：
  - 图片模型
  - 视频模型
  - `billing_mode=image` 的模型
  - MiMo 音频类模型
- Anthropic / Claude 文本模型恢复可在 Playground 对话助手中正常选择。

### 六、修正对话助手 key/model 串台

- 进一步排查确认：对话助手之前仍与其它 Playground 工具共用 `selectedKeyId`，会把其它区块的选择状态带进来。
- 已将对话助手拆成独立状态：
  - `chatSelectedKeyId`
  - `chatSelectedModel`
- 现在对话助手只使用自己的 API Key 与模型链路，不再受其它工具区块影响。
- 这次修复后，mimo / grok 等 key 切换时，对话助手模型列表会严格跟随当前 Key 的可用模型范围。

### 七、修复 Grok 账号表单误用 Anthropic 文案

- 排查确认：添加/编辑账号弹窗在选择 `Grok` 后，API Key 模式仍错误显示了 Anthropic 的默认值与提示文案。
- 具体表现为：
  - Base URL 仍回退为 Anthropic 默认地址
  - API Key placeholder 仍显示 `sk-ant-...`
  - 说明文案仍显示 Claude / Anthropic 相关提示
- 已补齐 `xai` 分支逻辑：
  - Grok API Key 模式默认 Base URL 改为 `https://api.x.ai`
  - API Key placeholder 改为 `xai-...`
  - 创建弹窗与编辑弹窗统一读取 xAI 专属提示文案
- 结果：
  - Grok 账号现在支持正常使用 API Key 模式添加与编辑
  - 不再误导为“只有 OAuth 才能使用”

### 八、修复 Grok 上游模型同步链路

- 排查确认：`xai + apikey + 第三方中转站` 在“同步上游支持的模型”时，后端错误将 `xai` 判定为 unsupported。
- 根因是同步逻辑只判断 `IsOpenAI()`，没有把 `xai` 视为 OpenAI-compatible。
- 已修复：
  - 上游模型同步逻辑改为走 `IsOpenAICompatible()`
  - `xai` 账号现在会正确按 OpenAI-compatible 模式请求上游 `/v1/models`
- 同时增强了错误透传：
  - 前端 toast 不再只显示固定文案
  - 现在会直接显示后端返回的安全错误摘要，便于定位 401 / 403 / 404 / 格式不兼容等问题

### 九、修复测试弹窗模型来源与白名单一致性

- 排查确认：账号“测试连接”弹窗的测试模型下拉，之前没有严格以本地白名单同步模型为准。
- 在 xAI/Grok 场景下，曾出现两类问题：
  - 白名单模型未进入下拉，导致测试模型为空
  - fallback 候选模型被直接列出，导致下拉里出现一批不该给客户测试的模型
- 已调整测试模型加载逻辑：
  - 先取后端返回候选模型
  - 再把本地白名单中存在但候选缺失的模型补入
  - 最终只保留白名单命中的模型作为测试下拉选项
- 结果：
  - `grok-4.5` 这类已同步到本地白名单的模型，可以在测试弹窗中直接选择
  - 不会再把无关的 fallback Grok 模型全量暴露到测试下拉里

### 十、修复 Grok API Key 测试误走 OAuth token 链路

- 排查确认：Grok `apikey` 账号在“测试账号连接”时，后端测试函数错误地统一读取 `access_token`，导致明明是 API Key 账号却报：
  - `No access token available`
- 已修复测试链路分支：
  - `OAuth` 账号继续走 `access_token`
  - `apikey` 账号改为读取 `api_key`
  - 缺少密钥时返回 `No API key available`
- 修复后已验证：
  - `grok-4.5` 可通过 `apikey` 账号成功完成测试请求
  - 测试结果能够正常返回上游响应内容

### 十一、当前 Grok 流量承接结论

- 已核实当前本地 `xai` 账号配置与调度状态：
  - `grok 备用`
    - `platform = xai`
    - `type = apikey`
    - `status = active`
    - `schedulable = true`
    - `model_mapping = { "grok-4.5": "grok-4.5" }`

    - `type = oauth`
    - `status = active`

  - 从模型支持上看，两者都声明支持 `grok-4.5`
  - 从当前真实调度上看，只有 `grok 备用` 会承接客户 `grok-4.5` 流量
  - `grok7.12 5日质保1` 需恢复为 `schedulable=true` 后，才会重新进入流量池
