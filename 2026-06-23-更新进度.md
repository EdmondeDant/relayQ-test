## 2. 根因结论

### 第一层根因：响应解析不兼容

上游并不是直接返回 `4xx/5xx`。

通过运行时证据确认，上游实际返回：

- HTTP `200`
- `Content-Type: text/event-stream`
- 事件中包含：
  - `response.output_text.delta`
  - `response.completed`

且图片结果不是原先已兼容的：

- `image_generation_call.result`

而是 Markdown 图片链接形态，例如：

```text
![generated](https://...)
```

所以旧逻辑会把“上游成功结果”误判成“没有图片输出”，最后向客户端返回 `502`。

### 第二层根因：图片编辑顶层模型选择错误

继续做真实客户视角复测后，又确认一个会直接影响客户可用性的点：

- `images/edits -> responses/editing` 转发时，顶层 `model` 曾被固定成 `gpt-5.4-mini`

但当前图片账号实际上只有：

- `gpt-image-2`

没有：

- `gpt-5.4-mini`

因此某些上游会直接返回 `403`，再被网关包装成泛化错误。

---

## 3. 本次代码修复

### 3.1 图片编辑响应解析修复

在 `responses` 非流式图片响应处理里新增 fallback：

- 当没有解析到 `image_generation_call.result` 时
- 从 `output_text` / Markdown 中提取图片 URL
- 下载图片字节
- 转成标准 OpenAI Images 输出：
  - `data[].b64_json`

### 3.2 图片编辑主模型选择修复

对 `images/edits -> responses/editing` 路径做收敛处理：

- 图片编辑请求的顶层 `model` 按当前图片模型处理
- 避免对仅支持图片模型的上游账号错误写入固定 orchestrator 模型

同时保留其他 `/responses image-only` 场景的原有语义，避免误伤。

### 3.3 回归扩展中顺手修复的后端问题

在扩大回归时，发现并修复了一个独立但真实的后端问题：

- `mode=image_generation` 时
- `output_cost_per_image_token` 不应再被错误合成到 `PerRequestPrice`

这个修复已纳入本次后端代码变更中。
